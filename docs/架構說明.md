# 架構說明

## 🏗️ 系統架構概述

本系統採用簡單的單體架構，由以下核心組件組成：

```
┌─────────────────┐
│   RSS Feed      │
│   (GCS API)     │
└────────┬────────┘
         │
         ↓ HTTP GET
┌─────────────────┐
│  fetch_news.py  │
│  (Orchestrator) │
└────────┬────────┘
         │
         ↓
┌─────────────────┐
│  rss_parser.py  │
│  (XML → JSON)   │
└────────┬────────┘
         │
         ↓
┌─────────────────┐
│   dedup.py      │
│  (去重檢查)      │
└────────┬────────┘
         │
         ↓
┌─────────────────┐
│  File System    │
│  (JSON 存儲)     │
└─────────────────┘
```

## 📦 核心組件

### 1. fetch_news.py（主調度器）

**職責：**
- 協調整個抓取流程
- 調用 RSS 解析器
- 調用去重工具
- 記錄執行日誌
- 錯誤處理

**輸入：** 無（定時觸發）

**輸出：**
- 原始 XML 檔案（`data/raw/`）
- JSON 新聞檔案（`data/processed/YYYY/MM/DD/`）
- 執行日誌（`data/fetch.log`）

**執行頻率：** 每日 5 次（09:00, 11:00, 13:00, 15:00, 17:00）

### 2. rss_parser.py（RSS 解析器）

**職責：**
- 從 URL 獲取 RSS feed
- 解析 XML 為 Python 對象
- 轉換為結構化 JSON
- 生成唯一新聞 ID
- 保存原始 XML 和 JSON

**輸入：** RSS URL

**輸出：**
- RSS XML 字符串
- 新聞條目列表（List[Dict]）
- JSON 檔案

**關鍵方法：**
```python
fetch()              # 獲取 RSS
parse_entries()      # 解析條目
save_raw_xml()       # 保存 XML
save_json()          # 保存 JSON
```

### 3. dedup.py（去重工具）

**職責：**
- 維護已處理新聞 ID 集合
- 檢查新新聞是否重複
- 更新去重數據庫

**輸入：** 新聞 ID 列表

**輸出：**
- 去重後的新聞 ID 列表
- 更新後的 `.processed_ids.json`

**去重邏輯：**
```python
news_id = MD5(link + title + published_date)
```

## 🗄️ 數據存儲

### 目錄結構

```
data/
├── raw/                      # 原始 RSS XML
│   └── rss_YYYYMMDD_HHMMSS.xml
├── processed/                # 處理後的 JSON
│   ├── .processed_ids.json   # 去重數據庫
│   └── YYYY/
│       └── MM/
│           └── DD/
│               └── {news_id}.json
├── fetch.log                 # 抓取日誌
└── cron.log                  # Cron 日誌
```

### JSON 格式

每則新聞的 JSON 結構：

```json
{
  "title": "新聞標題",
  "link": "https://...",
  "published": "Mon, 14 Apr 2026 10:00:00 GMT",
  "published_parsed": "2026-04-14T10:00:00",
  "summary": "新聞摘要...",
  "id": "abc123def456...",
  "fetched_at": "2026-04-14T14:13:00"
}
```

### 去重數據庫格式

```json
{
  "ids": ["abc123...", "def456...", ...],
  "count": 1234,
  "last_updated": "2026-04-14T14:13:00"
}
```

```cron
0 11 * * * cd /path/to/govmo-news && python src/backend/fetch_news.py
0 13 * * * cd /path/to/govmo-news && python src/backend/fetch_news.py
0 15 * * * cd /path/to/govmo-news && python src/backend/fetch_news.py
0 17 * * * cd /path/to/govmo-news && python src/backend/fetch_news.py
```

### 執行流程

```
Cron 觸發
    ↓
fetch_news.py 啟動
    ↓
記錄開始時間
    ↓
獲取 RSS
    ↓

    ↓

    ↓

    ↓

    ↓

```

## 🔒 安全考量

### 當前安全措施

1. **本地存儲** - 數據存儲在本地檔案系統
2. **無外部暴露** - 無 Web 伺服器，無 API 暴露
3. **只讀操作** - 僅從 GCS API 讀取，不寫入外部系統

## 📊 性能考量

### 當前假設

- **新聞數量：** 每日約 10-50 則新聞
- **執行時間：** 每次抓取 < 30 秒
- **存儲需求：** 每年約 10-20 MB（純文字 JSON）

### 瓶頸分析

| 組件 | 瓶頸 | 當前狀態 |
|------|------|---------|
| RSS 獲取 | 網絡延遲 | 可接受 |
| XML 解析 | CPU | 可接受 |
| JSON 寫入 | 磁碟 I/O | 可接受 |
| 去重檢查 | 記憶體 | 可接受 |

### 擴展性限制

如需處理更大規模：

1. **數據庫存儲** - 替換檔案系統為數據庫
2. **分散式抓取** - 多台機器同時抓取
3. **緩存機制** - 減少重複請求

## 📝 架構決策記錄

### 決策 1：使用檔案系統而非數據庫

**日期：** 2026-04-14

**決策：** 使用檔案系統存儲 JSON，不使用數據庫

**理由：**
- 簡單，無需安裝配置數據庫
- JSON 格式易於人類閱讀和調試
- 按日期組織便於管理
- 當前規模下性能足夠

**後果：**
- ✅ 優點：部署簡單，維護成本低
- ❌ 缺點：查詢效率低，不適合複雜檢索

**如需要改變：** 當需要複雜查詢或數據量超過 1GB 時考慮遷移到數據庫

### 決策 2：使用 cron 而非 Python 調度器

**日期：** 2026-04-14

**決策：** 使用系統 cron 而非 APScheduler 等 Python 庫

**理由：**
- Linux 標準工具，穩定可靠
- 無需額外依賴
- 易於監控和管理

**後果：**
- ✅ 優點：簡單，與系統整合好
- ❌ 缺點：跨平台支援差（Windows 需額外配置）
