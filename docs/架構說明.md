# 架構說明

## 🏗️ 系統架構概述

本系統採用簡單的單體架構，由以下核心組件組成：

```
┌─────────────────┐
│   RSS Feed      │
│   (GCS API)     │
└────────┬────────┘
         │
         ↓ HTTP GET
┌─────────────────┐
│  fetch_news.py  │
│  (Orchestrator) │
└────────┬────────┘
         │
         ↓
┌─────────────────┐
│  rss_parser.py  │
│  (XML → JSON)   │
└────────┬────────┘
         │
         ↓
┌─────────────────┐
│   dedup.py      │
│  (去重檢查)      │
└────────┬────────┘
         │
         ↓
┌─────────────────┐
│  File System    │
│  (JSON 存儲)     │
└─────────────────┘
```

## 📦 核心組件

### 1. fetch_news.py（主調度器）

**職責：**
- 協調整個抓取流程
- 調用 RSS 解析器
- 執行去重檢查
- 錯誤處理

**輸入：** 無（定時觸發）

**輸出：**
- JSON 新聞檔案（`data/processed/YYYY/MM/DD.json`）

**執行頻率：** 每日 5 次（09:00, 11:00, 13:00, 15:00, 17:00）

### 2. rss_parser.py（RSS 解析器）

**職責：**
- 從 URL 獲取 RSS feed
- 解析 XML 為 Python 對象
- 轉換為結構化 JSON（array 格式）
- 按日期組織存儲（每天一個 JSON 檔）
- 去重檢查（使用 RSS guid）

**輸入：** RSS URL

**輸出：**
- 新聞條目列表（List[Dict]）
- JSON 檔案（每天一個檔）

**關鍵方法：**
```python
fetch()              # 獲取 RSS
parse_entries()      # 解析條目
save_json()          # 保存 JSON（按日期分組）
```

## 🗄️ 數據存儲

### 目錄結構

```
data/
└── processed/                # 處理後的 JSON（提交到 Git，永久保存）
    └── YYYY/
        └── MM/
            └── DD.json       # 每天一個 JSON 檔（包含該日所有新聞）
```

### JSON 格式

每天的 JSON 檔包含新聞列表（array）：

```json
[
  {
    "title": "新聞標題",
    "link": "https://...",
    "guid": "https://...",
    "published": "2026-04-14T19:25:00+08:00",
    "published_parsed": "2026-04-14T11:25:00",
    "summary": "新聞摘要...",
    "fetched_at": "2026-04-14T19:29:12.610688"
  },
  ...
]
```

### 去重邏輯

使用 RSS feed 中的 `guid` 字段進行去重：
- 載入所有現有 JSON 檔
- 提取所有 guid 到集合
- 比對新新聞的 guid
- 跳過已存在的 guid


```
    ↓
```

## 📊 性能考量

### 當前假設

- **新聞數量：** 每日約 10-50 則新聞
- **執行時間：** 每次抓取 < 30 秒
- **存儲需求：** 每年約 10-20 MB（純文字 JSON）

### 瓶頸分析

| 組件 | 瓶頸 | 當前狀態 |
|------|------|---------|
| RSS 獲取 | 網絡延遲 | 可接受 |
| XML 解析 | CPU | 可接受 |
| JSON 寫入 | 磁碟 I/O | 可接受 |
| 去重檢查 | 記憶體 | 可接受 |

### 擴展性限制

如需處理更大規模：

1. **數據庫存儲** - 替換檔案系統為數據庫
2. **分散式抓取** - 多台機器同時抓取
3. **緩存機制** - 減少重複請求

## 📝 架構決策記錄

### 決策 1：使用檔案系統而非數據庫

**日期：** 2026-04-14

**決策：** 使用檔案系統存儲 JSON，不使用數據庫

**理由：**
- 簡單，無需安裝配置數據庫
- JSON 格式易於人類閱讀和調試
- 按日期組織便於管理
- 當前規模下性能足夠
- 數據提交到 Git，便於版本控制

**後果：**
- ✅ 優點：部署簡單，維護成本低，數據永久保存
- ❌ 缺點：查詢效率低，不適合複雜檢索

**如需要改變：** 當需要複雜查詢或數據量超過 1GB 時考慮遷移到數據庫

### 決策 2：使用 cron 而非 Python 調度器

**日期：** 2026-04-14

**決策：** 使用系統 cron 而非 APScheduler 等 Python 庫

**理由：**
- Linux 標準工具，穩定可靠
- 無需額外依賴
- 易於監控和管理

**後果：**
- ✅ 優點：簡單，與系統整合好
- ❌ 缺點：跨平台支援差（Windows 需額外配置）

### 決策 3：每天一個 JSON 檔

**日期：** 2026-04-14

**決策：** 每天一個 JSON 檔（array 格式），而非一則新聞一個檔

**理由：**
- 減少文件數量
- 便於按日期查閱
- 目錄結構更簡單（年/月）

**後果：**
- ✅ 優點：文件數量少，結構清晰
- ❌ 缺點：單日新聞過多時文件較大

### 決策 4：不保存原始 XML

**日期：** 2026-04-14

**決策：** 不保存原始 RSS XML，只保存處理後的 JSON

**理由：**
- 減少存儲空間
- JSON 已包含所有必要資訊
- 避免重複數據

**後果：**
- ✅ 優點：存儲效率高
- ❌ 缺點：無法追溯原始 XML 格式變化
